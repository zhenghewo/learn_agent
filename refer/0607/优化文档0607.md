### 图数据库

#### 总体架构

文档上传 / CDC 变更
│
▼
KnowledgeExtractAgent (LLM 抽取实体/关系)
│
▼
LangGraph 编排 (ingest / update 流水线)
│
├── store_vectors → ChromaDB
└── store_graph  → KnowledgeGraphService → Neo4j
│
▼
问答时 Graph 检索
├── QAAgent: LLM 生成 Cypher
└── GraphRAGPipeline: 实体链接 + 子图 + 路径

#### 连接与索引 — `KnowledgeGraphService`

- 启动时建 `name` / `type` / `source` 索引，加速实体定位与按来源删除。

#### 实体写入 — MERGE upsert

- 以 `name` 为唯一键，`MERGE` 保证幂等。
- 新建写全量属性；更新时仅覆盖非空 description，并递增 version。
- 增量更新场景通过 `source` 字段关联来源文件。

---

#### 关系写入 — 动态关系类型

- 先匹配头尾实体，再 `MERGE` 有向边。
- 关系类型由 LLM 抽取结果动态生成（如 `WORKS_AT`），边上存 confidence 和 source。

---

#### 图谱查询 — 子图遍历与通用 Cypher

- 从指定实体出发，遍历 1~N 跳内所有邻居，返回路径上的关系链与目标节点信息。
- 文档删除时，`DETACH DELETE` 同时移除节点及其关联边。

---

#### 写入链路 — LangGraph 入库编排

流程：`parse → extract → (store_vectors ∥ store_graph)`。抽取完成后，实体与关系并行写入 Neo4j，与向量入库互不阻塞。

---

#### 知识抽取 — 三元组来源

```
@dataclass
class Entity:
    name: str
    type: str
    description: str = ""

@dataclass
class Relation:
    head: str
    relation: str
    tail: str
    confidence: float = 0.0
```

- LLM 从文档块抽取 JSON 格式的 entities / relations。

- `_deduplicate` 跨 chunk 按 `name::type` 和 `(head, relation, tail)` 去重后输出。

---

#### 读取链路 A — QAAgent 动态 Cypher

- 查询改写后提取实体 → LLM 生成 1~2 条 Cypher → 执行并转为检索上下文。
- 与向量检索结果合并，图谱结果权重 ×1.2。

### 增量更新

#### 整体思路

不做全量重建，而是 CDC（变更捕获）→ 文档级处理 → 向量库/图谱局部更新。

变更检测 → 差量分析 → 增量解析 → 更新向量库 → 更新知识图谱

| 层级     | 策略                                                         |
| ---------- | -------------------------------------------------------------- |
| 检测     | 文件 SHA256 对比 / Watchdog 监听 / Kafka CDC                 |
| 向量库   | 按 `doc_id` 删旧 chunk，再写入新 chunk                   |
| 知识图谱 | `MERGE` upsert 实体 + 版本号；删除时按 `source` 清理 |

#### 新增

1. 新建文档（CREATE）

只处理这一个文件：parse → 对该文档的 chunk 做 embed → `add_chunks`。其他文档的向量不动。每个 chunk 有唯一 id：`{doc_id}#chunk-{index}`。

2. 已有文档追加/修改内容（MODIFIED）

不会只插入新增部分的向量。流程是：
（1）用 `doc_id` 删掉该文档的全部旧向量
（2）重新 parse 整篇文档
（3）把所有 chunk 重新 embed 并写入

#### 删除

核心：每个 chunk 带 `doc_id`，删除时按 `doc_id` 精确过滤。

1. 入库时绑定文档
   解析文档时，用文件路径生成稳定的 `doc_id`（路径 SHA256 前 16 位），每个 chunk 的 metadata 里都带上这个 `doc_id`。
2. 删除时按 `doc_id` 查再删
   文档删除（或修改前先清旧数据）时，用同一路径算出同一个 `doc_id`，在 Chroma 里 `where={"doc_id": doc_id}` 查出所有相关 chunk id，再批量 `delete`
